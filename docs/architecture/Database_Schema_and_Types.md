# Database Schema & Data Dictionary

## 1. Table Relationships
The DayLi database relies on Supabase PostgreSQL, employing strict Primary Key (PK) to Foreign Key (FK) constraints to ensure atomic cascading updates and Row Level Security (RLS). 

### Core Hierarchies
- **`profiles` (PK: `id`)**: The root user table. 
  - Tracks user metadata (`username`, `avatar_url`), setup flags (`is_work_setup_complete`), dietary rules, and work preferences.
  - Links 1-to-Many to `courses.user_id`, `work_shifts.user_id`, `custom_events.user_id`, and `expenses.user_id`.

- **`courses` (PK: `id`, FK: `user_id` -> `profiles.id`)**:
  - The parent object for the academic domain. Optionally links to a broader term grouping via `semester_id` (FK to `semesters.id`).

- **`classes` (PK: `id`, FK: `course_id` -> `courses.id`)**:
  - Bound specifically to a course. If the parent course is deleted, cascaded deletion automatically strips these block schedules from the unified dashboard calendar.

- **`assessments` (PK: `id`, FK: `course_id` -> `courses.id`)**:
  - Similar cascading setup. Inherits grade boundaries and links tracking metadata (`weight`, `my_score`) back to the specific course.

## 2. Advanced Data Types
Instead of flat SQL schemas, DayLi utilizes advanced Postgres types exposed optimally via the Supabase client:
- **String Arrays (`string[]`)**: 
  - Inside the `profiles` table, variables such as `allergies`, `secondary_cuisines`, and `days_off` are managed natively as string arrays. This allows the backend to perform strict inclusive/exclusive rule-checking (e.g., using `.contains('allergies', ['Peanuts'])`) directly within queries.
- **Enums & Strict Literals**:
  - The `assessments` table strictly types the `type` column to specific literal strings: `('assignment' | 'quiz' | 'project' | 'midterm' | 'final' | 'lab' | 'discussion' | 'exam' | 'other')`.
- **JSONB (`plan_data`)**:
  - Contained within `weekly_meal_plans`. Instead of creating 21 separate rows for a week's meals, the entire week payload generated by the AI Chef is stored as a single, massive JSONB cluster under `plan_data`. The structure maps internally (`{ "Monday": { "Breakfast": ..., "Lunch": ... } }`). PostgreSQL JSONB indexes ensure rapid unpacking on the frontend.

## 3. AI Caching Structure (`recipes` table)
To minimize the financial overhead of recursive token generation across Gemini 2.0 Flash models, the Meal Planning domain implements a Global Database Cache.

- **Global Shared Access**: The `recipes` table skips user-targeted RLS (or allows universally permissive reads) so all DayLi users query the exact same reference pool.
- **Schema Design**: 
  - `title` (Normalized and scrubbed string, used for strict matching or fuzzy searching).
  - `ingredients` (Structured array or JSON detailing quantities).
  - `instructions` (Formatted markdown steps).
  - `macro_estimations` (JSON dictionary indicating carb/protein breakdown).
- **The Execution**: When User A's meal engine says "Eat Avocado Toast", the system queries `recipes` where `title = 'Avocado Toast'`. If missing, the AI generates the instructions, charges the tokens, and upserts it. When User B gets "Avocado Toast" tomorrow, the app instantaneously pulls the cached `recipes` row, skipping Gemini completely. 

## 4. TypeScript Interfaces
The PostgreSQL schema strictly maps to the frontend via automatically generated Supabase types inside `src/types/supabase.ts`, preventing divergent structures.

### The Auto-Generated Core (`supabase.ts`)
- **`Database['public']['Tables']['*']`**: Defines exact `{ Row, Insert, Update }` primitives for every table (`profiles`, `courses`, `assessments`, `classes`, `work_shifts`, etc.).

### Domain Specific Abstractions (`courseDetails.ts`)
Instead of referencing deep paths through the app, domains export local aliases that point to the master Supabase index:
```typescript
export type Assessment = Database['public']['Tables']['assessments']['Row'];
export type Course = Database['public']['Tables']['courses']['Row'];
export type ClassSession = Database['public']['Tables']['classes']['Row'];
```

Furthermore, local UI structures mutate these database rows for Form/Display handling:
- **`MergedClassSession`**: Extends `ClassSession` by omitting `day_of_week` (which is singular in the DB) and instead substituting `days: string[]` to support modern UI toggles (e.g. allowing users to select "Mon/Wed/Fri" on one checkbox array).
- **`FormSession`**: A strictly local UI structure tracking temporary un-submitted data using JavaScript Date objects (`startTime: Date`) before mapping down to standard SQL string times upon persistence.
- **`DatedTimeSlot` (`worksDashboard.ts`)**: Extends the algorithmic `TimeSlot` interface to append a specific database-verified `shift_date`.
